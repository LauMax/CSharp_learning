# 1. 单一职责原则
每个类应该只有一个引起变化的原因 → 

一个类只承担一个职责，当一个类承担过多的职责，尤其是当拥有负责的内部状态时，修改一个功能可能会影响到内部状态，这种内部状态的变化往往会引起连锁反应，从而意外影响其他功能 

一个类功能过多，可维护性，可复用性，可扩展性都会下降

# 2. 开放封闭原则
软件实体应该对扩展开放，对修改封闭

在不破坏现有基础上，增加新功能，最大程度减少对现有代码的修改，减少错误的风险，提升系统整体稳定性

# 3. 里氏替换原则

子类应该能够替换他们的基类且不改变程序的执行结果

这意味着子类的对象，应该能在程序中代替其父类对象，而不引起任何错误或异常

这个原则确保了继承机制的正确性，当我们使用继承时，通常目的是为了代码复用，如果使用不当，很容易使得子类和父类行为不一致，导致在父类出现的任何场合中，子类不能无缝替换，从而使代码变得复杂和难以维护

# 4. 接口隔离原则

一个类不应该被迫依赖于他不需要的接口

也就是说：一个类应该仅实现它真正需要的接口中的方法

⇒ 将大的接口分解成更小的接口，更具体的接口，这样类就可以只关注他们需要的接口，避免实现不必要或无用的功能，从而减少系统的耦合度，增加代码的可维护性和灵活性

# 5. 依赖倒置原则

1. 高层模块不应该依赖于底层模块，他们都应该依赖于抽象
2. 抽象不应该依赖细节，而细节应该依赖抽象

在高层和底层之间加入抽象层，增加可维护性，灵活性

依赖注入是一种依赖倒置原则的实现，更易于测试，维护和扩展
